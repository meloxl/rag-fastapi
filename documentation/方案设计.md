# 恋爱知识 RAG 问答 — 方案设计

> 本文档描述系统架构、模块职责、数据流与关键设计决策，供后续扩展功能时给 AI 提供上下文。  
> 关联文档：[需求分析.md](./需求分析.md) | 运行说明：[../README.md](../README.md)

---

## 1. 设计目标

1. **简单可跑**：单仓库、本地 Ollama、无外部依赖，适合第一个 RAG 项目学习与演示。
2. **概念清晰**：与 Spring AI 的 DocumentReader / DocumentTransformer / DocumentWriter / Advisor 一一对应。
3. **改动可控**：新增能力时优先扩展对应模块，避免 `main.py` 膨胀。
4. **可验收**：问答响应携带 `sources`，便于核对检索是否命中正确篇章。

| 层级 | 选型 | 版本策略 |

## 4. 目录结构

```
rag-fastapi-local/
├── app/
│   ├── main.py              # 路由层：参数校验、HTTP 异常、编排调用
│   ├── config.py            # 环境变量与默认配置
│   ├── models.py            # Pydantic 请求/响应模型
│   └── rag/
│       ├── reader.py        # DocumentReader
│       ├── transformer.py   # DocumentTransformer
│       ├── writer.py        # DocumentWriter（Chroma）
│       └── chain.py         # 查询增强（Retrieval + Generation）
├── docs/                    # 【知识库】恋爱 Markdown 文档（会被 RAG 索引）
├── documentation/           # 【项目文档】需求/设计，不参与向量索引
├── chroma_db/               # Chroma 持久化（.gitignore）
├── requirements.txt
├── .env.example
└── README.md
```

> **注意**：`docs/` 与 `documentation/` 分离。`docs/` 下文件会被 `build_kb` 索引；项目说明类文档放在 `documentation/`，避免污染知识库。

## 5. 模块设计（对标 Spring AI）

### 5.1 DocumentReader — `app/rag/reader.py`

**职责**：从 `docs/` 读取所有 `.md` 文件，输出 LangChain `Document` 列表。

**核心函数**：

| 函数 | 说明 |
|------|------|
| `list_markdown_files()` | 列出知识目录下所有 `.md` 文件名 |
| `load_documents()` | 加载文档并填充 metadata |
| `sanitize_upload_filename()` | 上传文件名安全校验 |

**Metadata 约定**：

| 字段 | 来源 | 示例 |
|------|------|------|
| `source` | 文件名 | `恋爱常见问题和回答 - 单身篇.md` |
| `category` | 文件名解析 | `单身` / `恋爱` / `已婚` |
| `file_path` | 绝对路径 | 用于调试 |

**category 解析规则**（`_parse_category`）：文件名含「单身」「恋爱」「已婚」则映射对应分类，否则用文件名 stem。

---

### 5.2 DocumentTransformer — `app/rag/transformer.py`

**职责**：将整篇 Markdown 切分为语义完整的 chunk。

**实现**：`MarkdownHeaderTextSplitter`

```python
HEADERS_TO_SPLIT_ON = [
    ("#", "section"),      # 篇章级
    ("####", "question"),  # 单条问答
]
```

**设计理由**：知识库每条问答以 `####` 为标题，按标题切分可保证「问题 + 回答 + 推荐课程」在同一块或相邻块内，优于固定 500 字符盲切。

**切分后 metadata**：继承文件级 metadata，并追加 header 解析字段（如 `question`）。

---

### 5.3 DocumentWriter — `app/rag/writer.py`

**职责**：向量化并写入 / 加载 Chroma。

| 函数 | 说明 |
|------|------|
| `build_vector_store(chunks)` | **全量重建**：删除旧 `chroma_db/` 目录后 `Chroma.from_documents` |
| `load_vector_store()` | 加载已有持久化库 |
| `is_kb_built()` | 判断向量库目录是否存在且非空 |

**向量化实现**：委托 `app/rag/embeddings.py` 的 `get_embeddings()`，按 `EMBED_PROVIDER` 选择后端（见 5.5）。

**重建策略**：每次 `build_kb` 全量覆盖，避免重复 build 产生重复 chunk。适合当前小规模静态知识库；文档量大时可改为增量 upsert。

### 5.5 Embedding 工厂 — `app/rag/embeddings.py`

**职责**：统一创建 LangChain `Embeddings`，便于扩展多种向量化来源。

| EMBED_PROVIDER | 实现类 | 典型场景 |
|----------------|--------|----------|
| `ollama` | `OllamaEmbeddings` | 本地 `nomic-embed-text` 等 |
| `openai_compatible` / `siliconflow` | `OpenAIEmbeddings` + 自定义 `base_url` | 硅基流动 `Qwen/Qwen3-Embedding-8B`、OpenAI、其他兼容 API |

**扩展新厂商**：若为 OpenAI 兼容 `/v1/embeddings`，只需配置 `OPENAI_COMPATIBLE_API_BASE` + `EMBED_MODEL_ID`；若非兼容，在 `embeddings.py` 增加新 provider 分支。

**注意**：切换 provider 或模型后必须重新 `build_kb`，旧向量维度/空间不兼容。

---

### 5.4 查询增强 — `app/rag/chain.py`

**职责**：检索 → 拼接上下文 → LLM 生成 → 封装响应。

**流程**：

1. `load_vector_store()` 加载 Chroma
2. `as_retriever(search_kwargs={"k": RETRIEVE_K})` 默认 Top-3
3. `_format_context(docs)` 将检索结果格式化为带分类/来源/问题的上下文
4. `ChatPromptTemplate` + `ChatOpenAI(Qwen/Qwen3-VL-8B-Instruct)` + `StrOutputParser`
5. 返回 `AskResponse(question, answer, sources)`

**Prompt 设计要点**（system 消息）：

- 角色：专业恋爱关系顾问
- 约束：仅依据上下文；无则明确说明；不虚构链接

**sources 结构**（`SourceItem`）：

```json
{
  "category": "恋爱",
  "source": "恋爱常见问题和回答 - 恋爱篇.md",
  "preview": "争吵发生时，先冷静情绪..."
}
```

## 6. API 设计

| 方法 | 路径 | 请求 | 响应 | 说明 |
|------|------|------|------|------|
| GET | `/health` | — | `HealthResponse` | 存活探测 + 建库状态 |
| POST | `/build_kb` | — | `BuildKbResponse` | ETL 全流程：读 → 切 → 写 |
| POST | `/ask` | `{"question": "..."}` | `AskResponse` | RAG 问答 |
| POST | `/upload` | `multipart/form-data` file | `UploadResponse` | 保存 `.md` 到 `docs/` |

**错误码约定**：

| 场景 | HTTP | detail |
|------|------|--------|
| 无 markdown 文档 | 400 | 知识目录中没有 .md 文档 |
| 未建库就提问 | 400 | 请先调用 POST /build_kb |
| 上传非法文件 | 400 | 文件名/格式校验失败 |
| Ollama 不可用 | 500 | 问答失败：... |

**Swagger**：启动后访问 `http://127.0.0.1:8000/docs`

## 7. 配置设计 — `app/config.py`

| 环境变量 | 默认值 | 说明 |
|----------|--------|------|
| `DOCS_DIR` | `{项目根}/docs` | 知识库目录 |
| `CHROMA_DIR` | `{项目根}/chroma_db` | 向量库持久化 |
| `EMBED_PROVIDER` | `ollama` | 向量化后端：`ollama` / `openai_compatible` |
| `EMBED_MODEL_ID` | `nomic-embed-text` | 向量模型 ID（如 `Qwen/Qwen3-Embedding-8B`） |
| `OPENAI_COMPATIBLE_API_BASE` | `https://api.siliconflow.cn/v1` | 外部 Embedding API 地址 |
| `OPENAI_COMPATIBLE_API_KEY` | — | 外部 API Key（勿提交仓库） |
| `LLM_MODEL` | `Qwen/Qwen3-VL-8B-Instruct` | OpenAI 兼容 Chat 生成模型 |
| `LLM_API_BASE` | `OPENAI_COMPATIBLE_API_BASE` | 生成模型 API 地址 |
| `LLM_API_KEY` | `OPENAI_COMPATIBLE_API_KEY` | 生成模型 API Key |
| `RETRIEVE_K` | `3` | 检索返回条数 |
| `LLM_TEMPERATURE` | `0` | 生成温度，0 偏 deterministic |

复制 `.env.example` 为 `.env` 即可覆盖，无需改代码。

## 8. 数据流时序

### 8.1 构建知识库（build_kb）

```
docs/*.md
  → reader.load_documents()          # List[Document]
  → transformer.split_documents()    # List[Document] chunks
  → writer.build_vector_store()      # Chroma + persist chroma_db/
  → 返回 chunk_count
```

### 8.2 问答（ask）

```
用户 question
  → writer.load_vector_store()
  → retriever.invoke(question)       # Top-K chunks
  → chain: context + prompt + LLM
  → AskResponse(answer, sources)
```

### 8.3 上传（upload）

```
UploadFile (.md)
  → sanitize_upload_filename()
  → 写入 DOCS_DIR/{filename}
  → 提示用户手动 POST /build_kb
```

## 9. 关键设计决策记录（ADR 摘要）

| 决策 | 选项 | 最终选择 | 理由 |
|------|------|----------|------|
| 架构拆分 | 单文件 vs 四模块 | **四模块** `app/rag/*` | 对标 Spring AI，便于扩展 |
| 切分策略 | 固定字数 vs 标题切分 | **MarkdownHeaderTextSplitter** | 文档为 Q&A 结构 |
| 建库时机 | 启动自动 vs 手动 API | **手动 `/build_kb`** | 便于观察 ETL、控制 Ollama 调用 |
| 上传后建库 | 自动 vs 手动 | **手动重建** | 避免误覆盖；行为可预期 |
| LLM 接入 | 云端 API vs Ollama | **Ollama 本地** | 无 API Key、隐私友好 |
| 问答接口 | GET vs POST | **POST `/ask`** | 避免中文 URL 编码问题 |
| 项目文档位置 | 放 `docs/` vs 独立目录 | **`documentation/`** | 避免被 RAG 索引 |

## 10. 与参考项目的关系

[LangChain-RAG-FastAPI-Service](https://github.com/RMA-MUN/LangChain-RAG-FastAPI-Service) 能力对比：

| 能力 | 参考项目 | 本项目 V1 |
|------|----------|-----------|
| FastAPI 路由 | ✅ 多路由模块化 | ✅ 精简 4 接口 |
| RAG 核心 | ✅ app/rag | ✅ app/rag 四文件 |
| 向量库 | Chroma | Chroma |
| 用户系统 | Django + MySQL | ❌ |
| 会话持久化 | ✅ | ❌ |
| 前端 | Vue | ❌ |
| 文档格式 | 多格式 | 仅 Markdown |

**借鉴点**：分层命名（reader / transformer / writer / chain）、Chroma 持久化、领域 Prompt。  
**刻意省略**：微服务、用户态、会话 DB，降低第一个 RAG 项目的认知负担。

## 11. 扩展接入指南（给 AI 的修改地图）

扩展某能力时，优先修改下表对应文件，避免跨层耦合：

| 想加的能力 | 主要修改文件 | 注意事项 |
|------------|--------------|----------|
| 新文档格式（PDF） | `reader.py` | 扩展 loader 分支，metadata 保持一致 |
| 新切分策略 | `transformer.py` | 可组合多个 splitter |
| 换向量库（FAISS/PGVector） | `writer.py` | 保持 `build/load/is_built` 接口 |
| 换向量化模型/厂商 | `embeddings.py` + `.env` | 切换后重建 `chroma_db` |
| 改 Prompt / 加引用格式 | `chain.py` | 不动路由层 |
| 新 API 端点 | `main.py` + `models.py` | 业务逻辑仍下沉到 `rag/` |
| 按 category 过滤 | `chain.py` retriever | 使用 Chroma metadata filter |
| 上传后自动建库 | `main.py` upload | 调用 `build_kb` 逻辑抽成 service 函数 |
| 配置项 | `config.py` + `.env.example` | 同步更新 README |

**推荐新增模块**（当功能变复杂时）：

```
app/rag/
├── retriever.py   # 混合检索、rerank
├── memory.py      # 会话历史
└── service.py     # build_kb / ask 编排，供 main 调用
```
