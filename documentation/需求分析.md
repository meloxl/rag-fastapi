# 恋爱知识 RAG 问答 — 需求分析

> 本文档沉淀项目背景、功能边界与扩展方向，供后续迭代与 AI 辅助开发时参考。  
> 关联文档：[方案设计.md](./方案设计.md) | 运行说明：[../README.md](../README.md)

---

## 1. 项目背景

这是团队的**第一个 RAG（检索增强生成）应用**，目标是在标准 RAG 流程基础上做**适度简化**，快速落地一个可演示、可扩展的本地知识库问答系统。

**业务场景**：基于本地 Markdown 知识库，提供 **AI 恋爱知识问答**服务，覆盖单身、恋爱中、已婚三类典型关系阶段。

**参考项目**：[LangChain-RAG-FastAPI-Service](https://github.com/RMA-MUN/LangChain-RAG-FastAPI-Service/tree/base-rag)（企业级 RAG 对话服务）。本项目**不照搬**其 Django 用户体系、MySQL 会话、前端等复杂能力，仅借鉴「Reader / Transformer / Writer / 检索链」的分层思路。

---

## 2. RAG 流程：标准版 vs 本项目简化版

| 标准 RAG 步骤 | 本项目简化步骤 | 说明 |
|---------------|----------------|------|
| 1. 文档收集和切割 | **1. 文档准备** | 知识以 Markdown 形式存放在 `docs/`，结构尽量规范 |
| 2. 向量转化和存储 | **2. 文档读取** + **3. 向量转化和存储** | 读取与切分分开实现，向量写入 Chroma |
| 3. 切片过滤和检索 | （合并进步骤 4） | 第一版使用向量 Top-K 检索，不做混合检索/重排序 |
| 4. 查询增强和关联 | **4. 查询增强** | Retriever + 领域 Prompt + 本地 LLM 生成回答 |

## 4. 知识库需求

### 4.1 内置文档

项目已内置 3 篇 Markdown，位于 `docs/`：

| 文件 | 分类 | 内容 |
|------|------|------|
| `恋爱常见问题和回答 - 单身篇.md` | 单身 | 魅力提升、社交结识、线上交友、焦虑克服、相亲评估等 |
| `恋爱常见问题和回答 - 恋爱篇.md` | 恋爱 | 争吵处理、浪漫惊喜、保持自我、未来规划、缺点沟通等 |
| `恋爱常见问题和回答 - 已婚篇.md` | 已婚 | 工作家庭平衡、婚后浪漫、矛盾化解等 |

### 4.2 文档格式约定

- 格式：**Markdown**
- 篇章标题：`# 恋爱常见问题和回答 - xxx篇`
- 单条问答：`#### 问题标题` + 正文回答 + 可选「推荐课程」链接
- 切分策略依赖上述标题层级，**新增文档应遵循同一结构**，以保证检索 chunk 语义完整

### 4.3 文档扩展方式

- 手动放入 `docs/` 目录
- 通过 API `POST /upload` 上传 `.md` 文件
- **上传或增删文档后，必须调用 `POST /build_kb` 重建向量库**

## 5. 功能需求（V1 已确认）

### 5.1 必须实现

| ID | 功能 | 描述 | 验收标准 |
|----|------|------|----------|
| F1 | 健康检查 | `GET /health` | 返回服务状态、是否已建库、文档数、模型配置 |
| F2 | 构建知识库 | `POST /build_kb` | 读取 `docs/*.md`，切分、向量化，持久化到 `chroma_db/` |
| F3 | 恋爱问答 | `POST /ask` | 输入问题，返回 `answer` + `sources`（分类、文件名、摘要） |
| F4 | 文档上传 | `POST /upload` | 支持上传 `.md` 到 `docs/`，提示用户重建知识库 |

### 5.2 问答行为要求

- 角色定位：**专业恋爱关系顾问**
- 仅依据检索到的知识库内容回答，**不得编造**
- 知识库无相关内容时，明确回复「知识库中未找到相关内容」
- 回答语气：专业、温和、实用

### 5.3 明确不做（V1 范围外）

- 用户注册 / 登录 / 权限
- 多轮会话记忆
- PDF / Word 等非 Markdown 格式解析
- 混合检索（BM25 + 向量）、重排序
- 上传后自动重建向量库（当前为手动触发，降低意外覆盖风险）
- Web UI 前端
- 云端 LLM API（当前仅 Ollama 本地模型）

## 6. 非功能需求

| 类别 | 要求 |
|------|------|
| 部署 | 本地运行，依赖 Ollama，无需外部 API Key |
| 隐私 | 知识库与向量库均存本地 |
| 可维护 | 模块按 Reader / Transformer / Writer / Chain 拆分，便于对标 Spring AI 概念扩展 |
| 可配置 | 模型名、路径、检索条数等通过环境变量或 `.env` 配置 |
| Python | **3.11**（`pyproject.toml` 约束 `>=3.11,<3.12`，与 FastAPI 默认环境一致） |

## 7. 技术约束（已确认）

| 决策项 | 确认结果 |
|--------|----------|
| Web 框架 | FastAPI |
| RAG 编排 | LangChain |
| 向量库 | Chroma（持久化目录 `chroma_db/`） |
| Embedding | Ollama `nomic-embed-text` 或 OpenAI 兼容 API |
| LLM | OpenAI 兼容 Chat API `Qwen/Qwen3-VL-8B-Instruct` |
| 目录结构 | `app/rag/` 四模块拆分 |
| API | `health` + `build_kb` + `ask` + `upload` |

## 8. 验收用例

| 序号 | 输入问题 | 期望 |
|------|----------|------|
| 1 | 如何在社交场合主动结识心仪异性？ | 回答贴合单身篇；`sources.category` 含「单身」 |
| 2 | 恋爱中如何有效处理双方的争吵？ | 回答贴合恋爱篇；`sources.category` 含「恋爱」 |
| 3 | 婚后如何平衡工作与家庭责任？ | 回答贴合已婚篇；`sources.category` 含「已婚」 |
| 4 | 未建库时调用 `/ask` | 返回 400，提示先 `build_kb` |
| 5 | 上传新 `.md` 后不 `build_kb` 直接提问 | 仍使用旧向量库（行为符合设计） |

## 9. 后续扩展建议（供 AI / 开发者参考）

扩展时请优先保持现有模块边界，在对应层增量修改：

| 优先级 | 方向 | 建议落点 |
|--------|------|----------|
| P1 | 上传后可选自动 `build_kb` | `app/main.py` upload 路由 |
| P1 | 按 `category`  metadata 过滤检索 | `app/rag/chain.py` retriever |
| P2 | 支持 PDF（PyPDFLoader） | `app/rag/reader.py` |
| P2 | 多轮对话记忆 | 新增 `app/rag/memory.py` + 会话 ID |
| P2 | 混合检索 + Rerank | `app/rag/retriever.py` |
| P3 | Web 聊天界面 | 独立 `frontend/` 或 Streamlit |
| P3 | 切换 OpenAI / 其他云端 LLM | `app/config.py` + `app/rag/chain.py` |
| P3 | 对接参考项目的 Django 用户服务 | 独立微服务，不在此仓库强耦合 |

## 10. 术语表

| 术语 | 含义 |
|------|------|
| RAG | Retrieval-Augmented Generation，检索增强生成 |
| Chunk | 文档切分后的文本块，向量检索的最小单元 |
| Embedding | 文本向量表示，用于语义相似度检索 |
| Ollama | 本地运行开源 LLM / Embedding 模型的工具 |
| Chroma | 开源向量数据库，支持本地持久化 |
